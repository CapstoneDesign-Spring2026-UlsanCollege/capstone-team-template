# Week 9 Class Plan: Prove the MVP Is Real

## Purpose

This week starts **Sprint 3: MVP Verification + Code Ownership**.

The goal is no longer “build more stuff.”

The goal is to prove your app is real:

- it runs,
- your team understands it,
- the core flow is stable,
- bugs are visible,
- AI-assisted work is reviewed,
- progress is linked in GitHub.

## Main Rule

AI is allowed.

But AI-assisted work only counts if your team can:

1. run it,
2. explain it,
3. test it,
4. debug it,
5. link evidence for it.

If no one can explain the code, it is **Not Yet**.

## Weekly Progress Demo

Each team gets **5 minutes**.

| Segment | Time | Speaker |
|---|---:|---|
| What changed | 45 sec | PM |
| Show one improvement | 2 min | Demo Driver |
| Show evidence links | 1 min | QA Lead / Scribe |
| Explain one part | 1 min | Random member |
| Ask / blocker | 15 sec | PM |

No slides required unless they help.

Show the app, repo, Issues, PRs, board, docs, screenshots, tests, or CI.

## Good Demo Example

> We fixed the login empty-password bug. Here is Issue #18, PR #22, and a screenshot showing the new error message. Mina can explain the validation function.

## Weak Demo Examples

- “We worked on login.”
- “Almost done.”
- “AI made it.”
- “It works on my laptop.”
- “We forgot to push.”

## GitHub Board Cleanup

Use these columns:

- To Do
- Doing
- Done
- Blocked
- Nice Later

Rules:

- Every active card has one owner.
- Every active card has a Definition of Done.
- Every active card links to evidence or explains what evidence is needed.
- Bugs should become Issues.
- New features go to **Nice Later** unless the core MVP flow is stable.

## Individual Receipts

Each student must comment on the Sprint Packet with **2–3 links**.

Good receipts:

- PR link
- Issue link
- review comment
- CI run
- screenshot
- test result
- docs update
- demo video

Bad receipts:

- “I worked on it.”
- “We talked about it.”
- “AI helped.”
- “Almost done.”

## What Good Looks Like

A strong Week 9 team can say:

> Our core MVP flow works from login to dashboard to main feature. We have evidence links. We know which files each member can explain. We also found two confusing AI-generated areas and created Issues to investigate them.

## Common Mistakes

- Adding new features before the core flow is stable.
- Saying “done” without evidence.
- Hiding code no one understands.
- Letting one student explain everything.
- Leaving bugs as verbal comments instead of GitHub Issues.
- Submitting screenshots without links.
- Forgetting individual receipts.
