# Sprint 3 Guide: MVP Verification + Code Ownership

## Sprint 3 Theme

**Weeks 9–11: MVP Verification + Code Ownership**

Main question:

> Does the MVP work, and can the team explain it?

This sprint is not about adding lots of new features.

This sprint is about proving the project is real.

## Sprint 3 Goals

By the end of Sprint 3, your team should be able to:

- [ ] Run the app.
- [ ] Demo the core MVP flow.
- [ ] Explain the main code areas.
- [ ] Identify confusing or risky code.
- [ ] Fix or document P0/P1 bugs.
- [ ] Show GitHub evidence.
- [ ] Explain how AI was used and reviewed.
- [ ] Prepare for stricter QA in Sprint 4.

## The Core Rule

AI-assisted work only counts if your team can:

1. run it,
2. explain it,
3. test it,
4. debug it,
5. link evidence for it.

No mystery code.

If no one can explain it, it is **Not Yet**.

## Week-by-Week Plan

| Week | Focus | Main Output |
|---:|---|---|
| Week 9 | AI Code Ownership Audit | Identify what works, what is broken, what is unclear, and who owns what |
| Week 10 | Copilot workflow + bug triage | Use AI safely from Issues and document P0/P1 bugs |
| Week 11 | MVP verification | Prove the core MVP flow works and can be explained |

## Required GitHub Work

During Sprint 3, your repo should include:

- `docs/AI_CODE_OWNERSHIP_AUDIT.md`
- Weekly Sprint Packet Issues
- Updated GitHub Project board
- Bug Issues with severity labels or notes
- PRs linked to Issues
- AI Use Notes in AI-assisted PRs
- Evidence links for demos, tests, screenshots, docs, or CI

## Weekly Progress Demo

Every class, each team shows one visible improvement.

You may show:

- working feature
- fixed bug
- improved test
- stable demo path
- cleaned/refactored code
- updated setup docs
- updated architecture docs
- AI code review result

You must also show evidence.

No evidence = not done.

## Bug Severity

| Severity | Meaning |
|---|---|
| P0 | Final demo cannot work |
| P1 | Core feature broken or unreliable |
| P2 | Important, but workaround exists |
| P3 | Polish or nice improvement |

## Definition of Done for Sprint 3

By the end of Sprint 3:

- [ ] Core MVP flow works.
- [ ] Core MVP flow has evidence.
- [ ] P0 bugs are fixed or clearly documented.
- [ ] P1 bugs are fixed, assigned, or documented with mitigation.
- [ ] Every student can explain at least one code/doc/test area.
- [ ] AI-assisted work has human review.
- [ ] Sprint Packet links all major evidence.
- [ ] Nice-to-have features are moved to Nice Later.

## AI Use Note for PRs

Every AI-assisted PR should include:

```md
## AI Use Note

- AI tool used:
- What AI helped with:
- What I personally checked or changed:
- How I tested it:
- One part I still do not fully understand:
```

## What Good Looks Like

Good Sprint 3 work:

- We verified the main user flow.
- We found bugs and created Issues.
- We cut or delayed unstable features.
- We updated docs.
- We used AI for small, reviewed changes.
- Every member has evidence receipts.
- The team can explain the project without guessing.

## Common Mistakes

- Adding features instead of stabilizing.
- Accepting AI code without reading it.
- Opening huge PRs.
- Not testing the demo path.
- Not linking evidence.
- Saying “almost done.”
- Letting only one student understand the project.
- Hiding confusing code.

## Instructor Check Questions

Be ready to answer:

1. What is your current MVP flow?
2. Can you run it today?
3. What works right now?
4. What is broken or unreliable?
5. What code does each member own?
6. What code does the team not fully understand yet?
7. What did AI help with?
8. What did humans check or change?
9. What is your highest-risk bug?
10. What are you cutting or moving to Nice Later?

## Sprint 3 Motto

> Build less mystery. Show more proof.
