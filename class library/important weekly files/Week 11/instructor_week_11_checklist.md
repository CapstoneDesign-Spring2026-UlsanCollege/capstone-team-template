## Track Placement

### A. MVP Recovery

Use if:

- app does not reliably run,
- core flow is unclear,
- team cannot explain major parts,
- evidence is missing,
- AI-generated code is mystery code.

Focus:

- simplify,
- fix core flow,
- remove broken scope,
- document setup,
- create visible bug list.

### B. MVP Stabilization

Use if:

- app mostly works,
- bugs exist,
- docs/tests are weak,
- deployment/demo path is fragile,
- team can explain some but not all major areas.

Focus:

- bug triage,
- testing,
- refactoring,
- deployment reliability,
- setup docs.

### C. MVP Plus

Use if:

- core flow works,
- team can explain the main code path,
- evidence is linked,
- P0/P1 bugs are absent or controlled.

Focus:

- usability,
- accessibility,
- performance,
- better docs,
- final demo polish.

## Questions to Ask During Demos

### Evidence

- Where is the Issue?
- Where is the PR?
- Where is the screenshot, video, test, or CI result?
- Is this linked in the Sprint Packet?

### MVP Flow

- What is the user trying to do?
- What is the start state?
- What is the success state?
- What breaks if you try it twice?

### Code Ownership

- Who can explain this file?
- What does this function/component/route do?
- What changed this week?
- How was it tested?

### AI Use

- Did AI help with this?
- What did AI suggest?
- What did a human check or change?
- What part is still confusing?

### Scope Control

- Is this required for the final demo?
- Is this core flow or Nice Later?
- What should you stop building?

## Intervention Triggers

Flag teams if:

- no Sprint Packet draft,
- no links,
- no Issue for active work,
- no student can explain major code,
- AI use is hidden or vague,
- repeated “almost done” claims,
- no individual receipts,
- P0/P1 bugs are ignored,
- new features are added while core flow is unstable.

## Useful Phrases

- “Show me the app. Show me the repo. Show me the evidence.”
- “Almost done is not a status.”
- “If it is not linked, it did not happen.”
- “This is not about blame. It is about visible ownership and evidence.”
- “Move that to Nice Later. Stabilize the core flow first.”
- “AI can help, but your team still owns the result.”

## End of Class Check

Before teams leave, confirm:

- [ ] Core flow Issue exists.
- [ ] Evidence is linked.
- [ ] Bug list is updated.
- [ ] Ownership map is started.
- [ ] Sprint Packet draft is started.
- [ ] First Week 12 QA task is written.
