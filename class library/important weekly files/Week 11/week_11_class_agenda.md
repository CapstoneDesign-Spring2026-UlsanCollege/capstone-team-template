# Week 11 Class Agenda

## Sprint 3 Review: MVP Verification

Today we are proving the MVP is real.

The goal is not to add more features first. The goal is to verify the core flow, show evidence, and make bugs visible.

## Class Goals

By the end of class, each team should have:

- one verified MVP core flow,
- evidence linked in GitHub,
- a bug list with severity,
- a code ownership map,
- a clear plan for Week 12 QA Day.

## Opening Routine - 15 minutes

1. Open your repo.
2. Open your GitHub Project board.
3. Open your current Weekly Sprint Packet draft.
4. Choose the core MVP flow you will verify today.
5. Identify the Demo Driver, QA Lead/Scribe, and AI Steward for today.

## Weekly Progress Demos - 5 minutes per team

Each team shows one visible improvement.

| Segment | Time | Speaker | What to show |
|---|---:|---|---|
| What changed | 45 sec | PM | 1-3 specific changes |
| Demo | 2 min | Demo Driver | Working flow, fix, test, doc, or improvement |
| Evidence check | 1 min | QA Lead/Scribe | PRs, Issues, screenshots, tests, docs |
| Ownership check | 1 min | Random member | Explain one file, function, bug, test, or doc |
| Ask/blocker | 15 sec | PM | One specific need |

## Mini-Lesson - MVP Verification

A feature is not done because it exists.

A feature is done when the team can:

1. run it,
2. demo it,
3. explain it,
4. test or verify it,
5. debug or document problems,
6. link evidence.

## Studio Work Block 1 - Verify the Core Flow

Create or update a GitHub Issue called:

```txt
Verify MVP core flow
```

In that Issue, define:

- the user,
- the start state,
- the main action,
- the success state,
- what evidence proves it works,
- what bugs or limitations remain.

## Studio Work Block 2 - Capture Evidence

Each team must add at least one evidence item:

- screenshot,
- video,
- PR link,
- Issue link,
- test result,
- CI result,
- docs update.

## Studio Work Block 3 - Bug Triage

Create or update bug Issues.

Use severity:

- **P0:** final demo cannot work
- **P1:** core feature broken or unreliable
- **P2:** important but workaround exists
- **P3:** polish or nice improvement

## Studio Work Block 4 - Code Ownership

Each student must own one explainable area:

- file,
- function,
- component,
- route/API endpoint,
- database model,
- bug,
- test,
- setup step,
- doc,
- AI-assisted change.

## End Checkpoint - Last 15 minutes

Before leaving, each team must answer:

1. What works?
2. What breaks?
3. Where is the evidence link?
4. Who can explain the core code path?
5. What is your first Week 12 QA task?
