# Sprint 3 Guide: MVP Verification + Code Ownership

## Purpose

Sprint 3 is not about adding as many features as possible.

Sprint 3 is about proving your MVP is real.

By the end of Sprint 3, your team should be able to:

- run the core app flow,
- explain where the core flow lives in the code,
- identify bugs and risky code,
- show GitHub evidence,
- explain how AI helped and what humans checked.

## Sprint 3 Weeks

| Week | Focus | Required evidence |
|---:|---|---|
| Week 9 | AI Code Ownership Audit | `docs/AI_CODE_OWNERSHIP_AUDIT.md` linked in Sprint Packet |
| Week 10 | Copilot workflow + bug triage | P0/P1 bugs created as Issues, one small verified fix |
| Week 11 | MVP verification | Core MVP flow demo + ownership map |

## Main Rule

AI-assisted work only counts if your team can:

1. run it,
2. explain it,
3. test it,
4. debug it,
5. link evidence.

## What To Create

- `docs/AI_CODE_OWNERSHIP_AUDIT.md`
- GitHub Issues for major bugs or confusing code
- updated GitHub Project board
- small PRs with AI Use Notes
- Weekly Sprint Packet every Friday

## Definition of Done for Sprint 3

- [ ] Core MVP flow works or broken parts are clearly documented.
- [ ] Core MVP flow has demo evidence.
- [ ] P0/P1 bugs are created as Issues.
- [ ] Every student owns one code/doc/test/bug area.
- [ ] AI-assisted work has human review notes.
- [ ] Weekly Sprint Packets include receipts.

## What Good Looks Like

> “We verified the login-to-dashboard flow. It works locally. Issue #18 tracks the empty-password bug. PR #22 fixes it. Mina can explain the login component, Joon can explain the route, and Hana can explain the manual test.”

## Common Mistakes

- Adding new features before fixing the core flow.
- Saying “it works” without evidence.
- Accepting AI code no one understands.
- Opening giant PRs.
- Forgetting individual receipts.
